## 1. Аудит безопасности → SECURITY REPORT 🔴

Проверено прямым запросом прав на функции + `get_advisors(security)`.

### 🔴 CRITICAL — утечка PII через незащищённые перегрузки RPC
В проде существуют ПАРЫ перегрузок: защищённая `(int, text)` с SHA256-гейтом и **незащищённая** `(int)` без гейта, доступная роли `anon`:

| Функция | Доступ anon | Secret-гейт | Что утекает |
|---------|:-----------:|:-----------:|-------------|
| `get_tour_reminders(integer)` | ✅ да | ❌ нет | имя, телефон, tg_chat_id, дата тура клиентов |
| `get_review_requests(integer)` | ✅ да | ❌ нет | имя, телефон, tg_chat_id прошедших туров |

Любой человек с публичным anon-ключом (он зашит в PWA, строка 2105) может вызвать
`POST /rest/v1/rpc/get_tour_reminders {"p_days_ahead": 3650}` и **выгрузить контакты всей клиентской базы**. Подтверждено: `has_secret_gate=false`, `anon_exec=true`.

### 🔴 CRITICAL — запись в память клиента доступна anon
`upsert_client_memory(uuid, …)` — `anon_exec=true`, без гейта. Зная `client_id` (UUID), любой может перезаписать предпочтения/интересы любого клиента. Бот вызывает её через service_role, так что отзыв прав у anon безопасен.

### 🟠 HIGH — `get_bookings_by_phone(text)` без аутентификации
`anon_exec=true`, без гейта. Перебором телефонов любой получает чужие брони (тур, дата, статус). Это публичный эндпоинт PWA (строка 4525) — телефон = слабая «аутентификация». IDOR/enumeration.

### 🟡 MEDIUM
- `is_admin()` доступна anon/authenticated (advisor WARN). Проверить, не используется ли в RLS как доверенный признак.
- `pg_trgm` установлен в схеме `public` (advisor WARN).
- Таблицы `action_history`, `partners`: RLS включён, но **нет ни одной policy** → при service_role работает, но для anon/authenticated доступ полностью закрыт (это ок, но стоит задокументировать намерение).
- Auth: отключена защита от утёкших паролей (HaveIBeenPwned) — advisor WARN.

### ✅ Что сделано правильно
- `update_client_stage` — только service_role (anon/auth = false). ✅
- `get_kote_context`, `bot_upsert_client`, `get_new_leads` — anon-доступ, но закрыты SHA256-гейтом. ✅
- `app_upsert_lead` — anon by design (публичная точка входа лида). ✅
- RLS включён на всех таблицах. ✅

> ⚠️ Эти исправления — в БД (production). Они вынесены в отдельный **План изменений** в конце отчёта и НЕ применялись автоматически.

## ✅ ИЗМЕНЕНИЯ В PRODUCTION БД — ПРИМЕНЕНО 18.06.2026 (migration 006)

Отозваны права `anon`/`authenticated`/`PUBLIC` на 3 незащищённые функции —
проверено: `anon_exec=false`, `auth_exec=false`, `service_role` сохранён,
защищённые `(int,text)`-перегрузки для n8n не тронуты:

```sql
REVOKE EXECUTE ON FUNCTION public.get_tour_reminders(integer)  FROM anon, authenticated, PUBLIC;  -- ✅
REVOKE EXECUTE ON FUNCTION public.get_review_requests(integer) FROM anon, authenticated, PUBLIC;  -- ✅
REVOKE EXECUTE ON FUNCTION public.upsert_client_memory(uuid,...) FROM anon, authenticated, PUBLIC; -- ✅
```

🟠 **Осталось (НЕ применено намеренно):** `get_bookings_by_phone(text)`. PWA
([app-hyper.html:4524](platform/app-hyper.html)) зовёт её напрямую с публичным
anon-ключом на экране «Мои брони». Простой REVOKE сломает живую функцию.
Корректное исправление — подтверждение владения телефоном (OTP) или проксирование
через бэкенд (service_role); это фича, а не хотфикс. Вынесено в Roadmap (7–30 дней).

## ⚠️ Исходный план (для истории)

Цель — закрыть 2 CRITICAL + 1 HIGH, ничего не сломав. Бот ходит через **service_role** (права сохраняются), n8n — через **anon + secret** (использует защищённые `(int,text)`-перегрузки). Поэтому отзыв прав у `anon`/`authenticated` на НЕзащищённые перегрузки безопасен.

```sql
-- 1. Закрыть незащищённые перегрузки (PII-утечка). DROP опасен (вдруг кто-то
--    зовёт), поэтому сначала REVOKE — обратимо и безопасно.
REVOKE EXECUTE ON FUNCTION public.get_tour_reminders(integer)  FROM anon, authenticated;
REVOKE EXECUTE ON FUNCTION public.get_review_requests(integer) FROM anon, authenticated;

-- 2. Память клиента: писать должен только сервер (бот через service_role).
REVOKE EXECUTE ON FUNCTION public.upsert_client_memory(uuid, text[], text, text, text, text, text, integer, boolean) FROM anon, authenticated;

-- 3. Брони по телефону: убрать у anon перебор чужих броней.
--    ⚠️ ПРОВЕРИТЬ: PWA (app-hyper.html:4525) зовёт её с anon-ключом напрямую.
--    Перед REVOKE нужно решить: либо PWA ходит через бэкенд (service_role),
--    либо добавить гейт/токен. Поэтому этот пункт — отдельно, после фронт-правки.
-- REVOKE EXECUTE ON FUNCTION public.get_bookings_by_phone(text) FROM anon, authenticated;
```

Пункты 1–2 безопасны к немедленному применению. Пункт 3 завязан на PWA — применять только после правки фронта (иначе сломается экран «мои брони»).

После применения 1–2 нужно обновить ложные утверждения о безопасности в `Desktop/CLAUDE.md` (там сейчас написано, что `upsert_client_memory` — только service_role, что неверно до фикса).
